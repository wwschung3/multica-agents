# Skill: Skill Creation Guide (create-skill)

Designed to efficiently create or optimize skills, prioritizing the avoidance of redundant development while keeping the output lightweight and functional.

## When to Use

- When you need to create a new skill (e.g., "create skill," "new skill," "write SKILL.md").
- When an existing skill needs optimization (disorganized structure, unclear triggers, unfocused description).
- When you are unsure whether to "extend an existing skill" or "create a brand new one."
- When a Minimum Viable Product (MVP) is required (default to `SKILL.md` only; add scripts only when necessary).

## Common Triggers

- create-skill
- skill create
- New skill
- Create skill
- Optimize skill
- Rewrite SKILL.md
- Reuse existing skill

## Execution Checklist

1. **Search for Similar Skills**
   - Search for existing capabilities based on keywords to avoid "reinventing the wheel."
   - Identify candidate skills and potential reuse points (inheritable triggers, structures, or examples).

2. **Select Creation Strategy**
   - `Extend Existing`: Use if an existing skill covers more than 60% of the requirements.
   - `Create New`: Use if existing skills do not match the core scenario.
   - `Cancel Creation`: If an existing skill fully meets the needs, stop creation and provide usage instructions.

3. **Collect Essential Information**
   - Skill Name (`name`, lowercase-kebab-case).
   - Scenario Description (`description`, focus strictly on "When to Use").
   - Triggers (3–8 phrases).
   - Output Target (Documentation only / Documentation + Scripts).

4. **Generate Files**
   - **Location**: Skills must be generated under the `working_directory/skills` path.
   - **Default Output**: `SKILL.md`.
   - **Optional**: Generate `*.js` or `*.sh` only if strictly necessary.
   - **Dependencies**: Include `package.json` only if the script requires specific dependencies.

5. **Provide Post-Creation Suggestions**
   - How to verify trigger effectiveness.
   - How to iterate (adding examples, negative cases, or boundary conditions).

## Output Standards

- **Prioritize Conciseness**: Generate only `SKILL.md` by default.
- **Focused Description**: The `description` should only define trigger conditions, not the execution flow.
- **Clear Structure**: Must include at least "When to Use," "Execution Checklist," and "Output Standards."
- **Reusable**: Favor extending existing logic over duplicate creation.
- **Verifiable**: Provide at least one set of trigger test cases.

## Best Practices

- **Search before building** to avoid overlapping functions.
- **Extend first**, then consider building new.
- **Docs first, scripts later.**
- **Triggers** should cover synonymous expressions.
- Perform a **trigger verification** after every modification.

### OpenClaw YAML Frontmatter Specification
Every skill should include a standard YAML frontmatter:

```yaml
---
name: skill-name
description: "Used in the scenario of..."
homepage: [https://example.com](https://example.com) (optional)
metadata: { "openclaw": { "emoji": "🎨", "requires": { "bins": ["node"] } } }
---

```

**Core Fields**:
- `name`: Skill name (lowercase, kebab-case).
- `description`: Usage scenario description (recommended under 50 words).
- `homepage`: Relevant homepage URL (optional).
- `metadata.openclaw`: OpenClaw metadata.
  - `emoji`: Skill emoji (e.g., 🎨, 📊, ⚡, etc.).
  - `requires`: Dependency requirements.
    - `bins`: Required binary files (e.g., `node`, `bash`, `curl`).
    - `node_modules`: Required npm packages.
    - `env`: Required environment variables.

**Examples**:
- Documentation Guide: `metadata: { "openclaw": { "emoji": "📚" } }`
- Node.js Skill: `metadata: { "openclaw": { "emoji": "⚡", "requires": { "bins": ["node"] } } }`
- Shell Skill: `metadata: { "openclaw": { "emoji": "🐚", "requires": { "bins": ["bash"] } } }`

## Quick Self-Check

- Is the `name` in lowercase-kebab-case format?
- Does the `description` only describe "when to use"?
- Have 3-8 triggers been provided?
- Is only `SKILL.md` generated by default?
- Is it clearly explained when script files are necessary?
